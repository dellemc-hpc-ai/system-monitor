# vLLM vs Ollama — 3-Case × 3-IOLength Benchmark Plan

## Goal

Benchmark vLLM (with/without TurboQuant) vs Ollama across 3 I/O length scenarios on an NVIDIA L4 22 GiB GPU.

## Context / Assumptions

- **GPU:** NVIDIA L4 22 GiB
- **Model:** `Meta-Llama-3.1-8B-Instruct` — pure Attention, no Mamba, compatible with TurboQuant
  - vLLM: `hugging-quants/Meta-Llama-3.1-8B-Instruct-AWQ-INT4` (HuggingFace AWQ, ~4.9 GB)
  - Ollama: `llama3.1:8b-gguf` (Q4_K_M GGUF, imported from local file, ~4.9 GB)
- **vLLM:** Docker `vllm/vllm-openai:latest` on port 8000
- **Ollama:** Running locally on port 11434 (already running)
- Ollama and vLLM cannot share GPU — must stop one before starting the other

## 3 Cases × 3 IOLengths Matrix

| Case | Engine | KV Cache |
|---|---|---|
| 1. vLLM + TurboQuant | vLLM Docker | `--kv-cache-dtype turboquant_k8v4` |
| 2. vLLM (fp8) | vLLM Docker | `--kv-cache-dtype fp8` |
| 3. Ollama | Ollama native | Q4_K_M GGUF, no TurboQuant |

| IOLength | ISL | OSL | Use Case |
|---|---|---|---|
| A | 400 | 400 | Translation simulation |
| B | 200 | 2000 | Text generation |
| C | 2000 | 200 | Summarization |

**Total: 9 benchmark runs** (3 cases × 3 IOLengths), each run includes 3 phases (Cold / Cache / Steady).

## Metrics

Per run per phase:
- Latency (ms) — wall-clock time
- Throughput (tok/s)
- GPU Memory (MiB) — measured before/after
- Completion tokens

## Key Risks

1. **Ollama vLLM GPU conflict** — must stop Ollama before vLLM starts; restart Ollama before Case 3
2. **vLLM no-TurboQuant OOM on L4** — fp8 KV cache may exceed 22GB with 2000 output tokens; reduce `--gpu-memory-utilization` to 0.85 or lower
3. **Qwen2.5-7B still broken** — won't use; sticking with Llama 3.1 8B AWQ which works on both engines
4. **Long runtime** — ~3-4 hours; use `notify_on_complete` for background runs
