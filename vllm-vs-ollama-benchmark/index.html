Key Finding: Cache provides zero speedup — prompt is too short

Cache Hit and Cold Start are nearly identical for both vLLM configs. The 35-token prompt KV is negligible compared to the 256-token generation. Cache shines when system prompts are long (RAG, multi-turn) — here it doesn't matter.

Key Finding: TurboQuant has a steep steady-state cliff — 109% slower under burst

TurboQuant starts fast (5.2s) but degrades to 10.8s under burst. fp8 is stable at 10.9s across all phases. For multi-user production servers, fp8 is the safer choice.

Analysis

FASTEST COLDvLLM + TurboQuant at 5,165ms — 2x faster than fp8 on cold-start inference. The 2.5x KV compression ratio reduces memory bandwidth pressure during generation.

STABLE UNDER LOADvLLM fp8 stays at 10.9s across all phases (4% degradation vs 109% for TurboQuant). Best choice for multi-user servers where burst load is common.

73% LESS VRAMOllama uses 5.5GB vs 20GB for vLLM. Good for memory-constrained environments or running alongside other workloads.

Why cache didn't help

The prompt is only ~35 tokens. Its KV footprint is dwarfed by the 256-token generation. Cache speedup only matters when the prompt KV is a significant fraction of total compute — i.e., long system prompts in RAG or multi-turn chat.

Why TurboQuant has a steady-state cliff

TurboQuant's 2.5x compression means more KV entries fit in the same 20GB — but each entry is more tightly packed. When the cache fills under burst load, eviction forces recomputation from scratch, which hits the memory bus harder than fp8's larger but more predictable entries.

Methodology note

Each phase measures a distinct cache state. Cold = pure inference (cache purged). Cache Hit = KV reuse. Steady = sustained burst. This separates concerns that were conflated in earlier mixed benchmarks.
